🤖 БОТЫ: structured logging + recoverable handlers

Цель: внедрить логирование и безопасные обработчики, чтобы бот не падал при ошибках пользователя/сервиса.

Связь с теорией дня 16: закрепляешь try/except, error context и recovery path для продакшн-бота.

Фокус на смысл: важна не «тишина ошибок», а управляемое поведение: лог, понятный ответ пользователю и продолжение работы.

Что нужно сделать (конкретно)

  1. Есть logger (уровни info/error).
  2. Командные handlers обёрнуты в safe-wrapper или try/except.
  3. Есть fallback ответ пользователю при исключении.
  4. Лог содержит context: user/chat/command/error.

Критерии готовности (Definition of Done)

  • Искусственно вызванная ошибка не роняет процесс бота.
  • В логе остаётся структурированная запись ошибки.
  • Пользователь получает дружелюбный fallback-ответ.

  • Код logger + safe handlers.
  • Фрагмент логов с примером ошибки и восстановления.

🕷️ ПАРСИНГ: resilient pipeline + retry

Цель: сделать пайплайн устойчивым к временным сбоям сети/структуры страницы и сохранить quality-report.

Связь с теорией дня 16: закрепляешь retry/backoff, error budget и частичное восстановление в парсинге.

Фокус на смысл: даже при сбое части шагов система должна давать полезный результат и прозрачный отчёт.

Что нужно сделать (конкретно)

  1. Есть retry wrapper для fetch/parse шагов.
  2. Есть классификация ошибок (temporary/permanent).
  3. Есть продолжение обработки после локальной ошибки.
  4. Есть итоговый report с failed/skipped/retried.

Критерии готовности (Definition of Done)

  • При временной ошибке срабатывает retry и фиксируется в логе.
  • При невосстановимой ошибке задача помечается корректно, без краша всего run.
  • Итоговый report отражает успешные и проблемные элементы.

  • Код retry/backoff и resilient-runner.
  • Логи/отчёт по прогону с минимум одной симулированной ошибкой.
