🤖 Задание #1 (CSV ЭКСПОРТ В БОТЕ)

Цель: довести бот до клиентского результата: админ нажимает команду и получает корректный CSV-файл.

Связь с теорией дня 6: закрепляешь экспорт как сервисную функцию: проверка доступа, формирование файла, отправка и безопасный cleanup.

Фокус на смысл: это «нажал кнопку — получил отчёт»: без ручной возни, с понятным результатом для заказчика.

Что нужно сделать (конкретно)

  1. /export закрыт guard-проверкой.
  2. CSV строится через DictWriter с явными колонками.
  3. Файл отправляется через answer_document.
  4. Есть обработка пустых данных и cleanup временного файла.

Критерии готовности (Definition of Done)

  • Не-админ не может вызвать экспорт.
  • Админ получает CSV с ожидаемыми колонками.
  • После отправки временный файл удаляется и не копится в проекте.

Что сдать на проверку

  • Код команды /export (guard + генерация CSV + отправка).
  • Лог/подтверждение, что файл отправлен и очищен после отправки.

🕷️ Задание #2 (EMAIL REPORT + RETRY + SCHEDULER)

Цель: автоматизировать доставку отчётов: парсер формирует сводку, отправляет её по email и устойчиво переживает временные сбои.

Связь с теорией дня 6: закрепляешь связку reporting + retry + scheduled run для регулярной ценности клиенту.

Фокус на смысл: ценность появляется не от «разового скрипта», а от регулярной и надёжной доставки отчётов без ручного контроля.

Что нужно сделать (конкретно)

  1. Собирается сводка total/filtered/new/top5.
  2. Есть send_email и send_with_retry.
  3. Есть метки времени запуска и лог финального статуса.
  4. Есть явная настройка планового запуска (комментарий/конфиг).

Критерии готовности (Definition of Done)

  • Email-отчёт собирается из реальных метрик, а не заглушек без структуры.
  • При временной ошибке отправки срабатывает retry-логика.
  • В коде явно зафиксирован способ планового запуска (cron/Task Scheduler/аналог).

Что сдать на проверку

  • Код функций send_email и send_with_retry.
  • Пример запуска с метриками и отметкой времени start/end.
