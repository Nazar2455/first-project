День 16 · 🕷️ ПАРСИНГ

Тема: делаем парсер устойчивым к сетевым сбоям и ошибкам данных через логи, retry и алерты.

← День 16 (выбор трека) Переключиться на БОТЫ

1) Что логировать по этапам

  • start / finish run
  • количество полученных и валидных строк
  • ошибки сети и парсинга
  • время выполнения

2) Retry на сетевые запросы

пример идеи
for attempt in range(1, 4):
    try:
        response = requests.get(url, timeout=20)
        response.raise_for_status()
        break
    except Exception as e:
        logger.warning(f"attempt {attempt} failed: {e}")
        if attempt == 3:
            raise

3) Алерты при критических падениях

Минимум: писать отдельный critical-лог. Лучше: отправлять уведомление на email/Telegram админу при фатальной ошибке.

Если скрипт падает ночью, ты должен узнать об этом раньше клиента.

4) Практика

  1. Добавь логирование этапов в parser.
  2. Реализуй retry на HTTP-запросы.
  3. Сымитируй падение и проверь alert/critical-log.
  4. Добавь раздел «Диагностика» в README.
