🤖 БОТЫ: task queue + worker

Цель: вынести тяжёлые действия из обработчика сообщений в очередь задач с контролем статуса.

Связь с теорией дня 21: закрепляешь асинхронный паттерн: enqueue, worker, retry и идемпотентность.

Фокус на смысл: пользователь должен получить быстрый ответ, а тяжёлая работа — уйти в фон и быть наблюдаемой.

Что нужно сделать (конкретно)

  1. Есть модель Task со статусами queued/running/done/failed.
  2. Есть worker_once/run_task.
  3. Есть retry/backoff и max_attempts.
  4. Есть task_status и idempotency key.

Критерии готовности (Definition of Done)

  • Дубликаты задачи не создаются при повторном запросе.
  • При ошибке задача либо ретраится, либо уходит в failed по правилам.
  • Статус задачи можно запросить отдельно.

🕷️ ПАРСИНГ: batch queue + dead-letter

Цель: собрать устойчивый batch-runner с отдельным dead-letter контуром для проблемных задач.

Связь с теорией дня 21: применяешь механики приоритизации, классификации ошибок и восстановления из DLQ.

Фокус на смысл: одна «плохая» задача не должна валить весь батч — у системы должен быть безопасный путь восстановления.

Что нужно сделать (конкретно)

  1. Есть JOBS/DEAD_LETTER.
  2. Есть run_jobs_once и классификация retriable/non-retriable.
  3. Есть priority сортировка.
  4. Есть requeue_dead_letter.

Критерии готовности (Definition of Done)

  • Runner обрабатывает пакет задач с разными приоритетами.
  • Невосстановимые ошибки уходят в dead-letter.
  • Есть сценарий возврата задачи из dead-letter в очередь.
