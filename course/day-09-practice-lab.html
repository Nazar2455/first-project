🤖 БОТЫ: деплой + smoke-check

Цель: перевести бота из локального режима в прод, где он стабильно работает и проверяется быстрым health/smoke-сценарием.

Связь с теорией дня 9: закрепляешь fail-fast конфиг, health-check, логирование инцидентов и доказательство рабочего деплоя.

Фокус на смысл: прод «живой» только когда есть не просто запуск, а контроль состояния и понятный путь восстановления.

Что нужно сделать (конкретно)

  1. Fail-fast по env (TOKEN).
  2. Есть /health.
  3. Есть логи start/crash/shutdown.
  4. Есть proof прода (URL/скрин/лог).

Критерии готовности (Definition of Done)

  • Бот стартует в проде и проходит smoke-check.
  • Ошибки не скрываются — есть логи падения и остановки.
  • Есть подтверждение рабочего деплоя (ссылка/лог/скрин).

Что сдать на проверку

  • Код entrypoint с проверкой env и /health.
  • Фрагмент логов старта и доказательство прод-запуска.

🕷️ ПАРСИНГ: scheduler + runbook

Цель: сделать парсер регулярным сервисом с наблюдаемостью и базовой аварийной реакцией.

Связь с теорией дня 9: закрепляешь автозапуск, мониторинговые метрики, алерты на фатал и runbook для восстановления.

Фокус на смысл: если запуск по расписанию не контролируется логами и алертами, клиенту нельзя гарантировать стабильный результат.

Что нужно сделать (конкретно)

  1. Есть job_start/job_end логи.
  2. Есть метрики total/filtered/new.
  3. Есть notify_admin на фатал.
  4. Есть cron/task-scheduler конфиг и runbook пункт.

Критерии готовности (Definition of Done)

  • После автозапуска в логах видны start/end и метрики.
  • При фатальной ошибке отправляется уведомление администратору.
  • Runbook описывает где смотреть логи и как перезапустить задачу.

Что сдать на проверку

  • Код job/scheduler + notify_admin.
  • Пример лога успешного прогона и одного аварийного сценария.
