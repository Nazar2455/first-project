Практика дня 7

🤖 Задание #1 (БОТ-КЕЙС: README + DEMO + ОТКЛИК)

Цель: превратить технический результат в кейс, который продаёт: чтобы клиент за 1–2 минуты понял пользу и следующий шаг.

Связь с теорией дня 7: закрепляешь упаковку кейса (pain → solution → result), демо-подачу и структуру отклика.

Фокус на смысл: сильный кейс = доказательство, что ты решаешь бизнес-задачу, а не просто «умеешь писать код».

Что нужно сделать (конкретно)

  1. README с блоками: проблема → решение → функции → запуск → результат.
  2. Короткая кейс-карточка (5–8 пунктов).
  3. Сценарий демо 60–90 секунд.
  4. Шаблон отклика с CTA и ссылкой на кейс.

Критерии готовности (Definition of Done)

  • README читается клиентом и даёт понятный эффект от решения.
  • Есть короткая версия кейса для чата и полная для портфолио.
  • Отклик заканчивается явным next step (демо/созвон).

Что сдать на проверку

  • Готовый текст README и кейс-карточки.
  • Сценарий демо + шаблон отклика.

🕷️ Задание #2 (PARSING-КЕЙС: МЕТРИКИ + ОГРАНИЧЕНИЯ + КП)

Цель: упаковать parsing-проект как услугу с понятной ценностью, рисками и коммерческим предложением.

Связь с теорией дня 7: закрепляешь метрики кейса, блок ограничений и две формы КП (короткая/развёрнутая).

Фокус на смысл: клиент покупает предсказуемый результат и прозрачные условия, а не «абстрактный скрипт».

Что нужно сделать (конкретно)

  1. Шаблон кейса с метриками (обработано/валидно/новых/время).
  2. Блок ограничений и рисков + меры контроля.
  3. Короткое КП для чата и развёрнутое КП.

Критерии готовности (Definition of Done)

  • Есть измеримые метрики последнего прогона.
  • Ограничения и риски сформулированы заранее и честно.
  • Обе версии КП содержат формат результата, сроки и next step.

Что сдать на проверку

  • Текст parsing-кейса с метриками и ограничениями.
  • Короткая и развернутая версия КП.
