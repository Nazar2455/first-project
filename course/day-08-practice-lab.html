🤖 Задание #1 (FAQ-БОТ V1)

Цель: собрать второй коммерческий бот с предсказуемой навигацией по FAQ и минимумом дублирования кода.

Связь с теорией дня 8: закрепляешь callback-архитектуру, единый handler и переиспользуемый формат данных FAQ.

Фокус на смысл: один универсальный handler упрощает поддержку и позволяет быстро добавлять новые FAQ без переписывания логики.

Что нужно сделать (конкретно)

  1. Inline клавиатура с callback faq:<key>.
  2. Один универсальный callback-handler.
  3. Справочник FAQ (словарь/JSON), минимум 3 раздела.
  4. Fallback для неизвестного callback + call.answer().

Критерии готовности (Definition of Done)

  • Новый FAQ-раздел добавляется в данные без создания нового handler.
  • Невалидный callback не ломает сценарий (есть fallback).
  • После каждого нажатия кнопки вызывается call.answer().

Что сдать на проверку

  • Код клавиатуры и универсального callback-handler.
  • Пример теста: валидный FAQ callback + неизвестный callback.

🕷️ Задание #2 (PARSER-PROJECT-2)

Цель: показать воспроизводимость: быстро адаптировать парсер под новый источник через конфиг селекторов.

Связь с теорией дня 8: закрепляешь адаптерный подход: SELECTORS + безопасный парсинг + валидация обязательных полей.

Фокус на смысл: ценность в том, что ты можешь запускать второй/третий парсер по шаблону, а не «с нуля» каждый раз.

Что нужно сделать (конкретно)

  1. Адаптер селекторов (SELECTORS для card/title/price/url).
  2. Безопасный парсинг через select_one + проверки.
  3. Валидация обязательных полей (title/url).
  4. Экспорт result2.csv и вывод количества строк.

Критерии готовности (Definition of Done)

  • Селекторы собраны централизованно и легко меняются.
  • Неполные карточки не попадают в финальную выгрузку.
  • Есть файл result2.csv и метрика количества строк.

  • Код SELECTORS + функции парсинга/валидации.
  • Лог запуска с итогом по строкам.
