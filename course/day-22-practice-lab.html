🤖 БОТЫ: DTO + validation boundary

Цель: сделать единый и безопасный вход данных в систему бота.

Связь с теорией дня 22: закрепляешь контракты данных: DTO, нормализация и управляемые ошибки.

Фокус на смысл: грязные входные данные должны превращаться либо в валидный объект, либо в понятную пользователю ошибку.

Что нужно сделать (конкретно)

  2. Есть validate_lead + нормализация.
  3. Есть ERROR_MAP/to_user_error.
  4. Есть единый ответ handle_lead_input.

Критерии готовности (Definition of Done)

  • Невалидные данные не проходят дальше boundary слоя.
  • Пользователь получает единообразные сообщения об ошибках.
  • Есть логи валидных и невалидных кейсов.

🕷️ ПАРСИНГ: schema-check + quarantine

Цель: отделить корректные строки от проблемных и контролировать качество данных.

Связь с теорией дня 22: реализуешь quality-gate для парсинга через валидацию схемы и quarantine.

Фокус на смысл: плохие строки не должны ломать поток — они изолируются, а отчёт показывает качество.

Что нужно сделать (конкретно)

  1. Есть REQUIRED_FIELDS и validate_row.
  2. Есть split_valid_invalid.
  3. Есть quarantine invalid rows.
  4. Есть quality report total/valid/invalid.

Критерии готовности (Definition of Done)

  • По итогам прогона есть valid и invalid части.
  • У каждой invalid строки есть reason.
  • Quality-report отражает долю валидных данных.
