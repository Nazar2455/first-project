🔍 Как читать обработчики и не путаться

У обработчиков есть простая модель: событие → фильтр → функция → ответ.

  1. Событие: пришло сообщение или callback.
  2. Фильтр: подходит ли это событие под условие.
  3. Функция: что мы делаем (сохраняем, считаем, отвечаем).
  4. Ответ: что видит пользователь.

Мини-пример чтения обработчика «по шагам»

Разбор структуры хендлера
@router.message(Command("start"))
async def cmd_start(message: Message):
    await message.answer("Привет!")
  • Событие: пришло сообщение.
  • Фильтр: это именно команда /start.
  • Функция: вызывается cmd_start.
  • Ответ: пользователь получает «Привет!».
Если читать любой хендлер именно в таком порядке, код становится понятным даже новичку.

🤖 Интерфейс бота: зачем нужны кнопки

Если пользователь видит только текст и команды, он теряется. Кнопки снижают трение и увеличивают вероятность, что бот реально будут использовать.

Простыми словами: ты строишь не «чат с командами», а маленький интерфейс приложения внутри Telegram.

Reply клавиатура (основное меню)

keyboards/main_kb.py
from aiogram.types import KeyboardButton, ReplyKeyboardMarkup

main_reply_kb = ReplyKeyboardMarkup(
    keyboard=[
        [KeyboardButton(text="📝 Оставить заявку")],
        [KeyboardButton(text="❓ FAQ"), KeyboardButton(text="📞 Контакты")]
    ],
    resize_keyboard=True,
    input_field_placeholder="Выбери действие"
)

Что тут важно понимать по строкам

  • keyboard=[[...], [...]] — каждая вложенная строка это отдельный ряд кнопок.
  • resize_keyboard=True — Telegram делает клавиатуру компактнее и удобнее.
  • input_field_placeholder — подсказывает пользователю, что делать дальше.

Подключение клавиатуры в /start

handlers/start.py
from aiogram import Router
from aiogram.filters import Command
from aiogram.types import Message
from keyboards.main_kb import main_reply_kb

router = Router()

@router.message(Command("start"))
async def cmd_start(message: Message):
    await message.answer(
        "Привет! Я помогу принять заявку. Выбери действие ниже:",
        reply_markup=main_reply_kb
    )

Логика простая: при входе в бота ты сразу даёшь человеку понятные варианты действий, вместо «угадай, какую команду ввести».

↔ Reply vs Inline — когда что использовать

  • Reply: большие постоянные действия (главное меню).
  • Inline: действия внутри конкретного сообщения (например, «Подтвердить», «Отмена», «Подробнее»).

Мини-правило выбора

Если действие относится ко всему боту — обычно Reply. Если действие относится к одному сообщению или шагу — обычно Inline.

Inline пример

keyboards/main_kb.py — inline
from aiogram.types import InlineKeyboardButton, InlineKeyboardMarkup

lead_inline_kb = InlineKeyboardMarkup(
    inline_keyboard=[
        [InlineKeyboardButton(text="🚀 Начать заявку", callback_data="lead:start")],
        [InlineKeyboardButton(text="📚 FAQ", callback_data="faq:open")]
    ]
)
Принцип: callback_data должен быть короткий и предсказуемый. Например, формат `entity:action`.
🔥 Callback = диалог с ботом
«Я нажал кнопку, и что? Бот тупо говорит в ответ?» Нет! Callback — это когда каждая кнопка ведёт сценарий дальше. Пользователь нажимает, состояние меняется, появляются новые кнопки. Это не чат, это приложение! 📱

🧩 Callback-обработчики без хаоса

handlers/callbacks.py
from aiogram import Router, F
from aiogram.types import CallbackQuery

router = Router()

@router.callback_query(F.data == "lead:start")
async def lead_start_callback(call: CallbackQuery):
    await call.message.answer("Окей, начинаем заполнение заявки 👇")
    await call.answer()

@router.callback_query(F.data == "faq:open")
async def faq_open_callback(call: CallbackQuery):
    await call.message.answer("FAQ:\n1) Сроки\n2) Стоимость\n3) Поддержка")
    await call.answer()

Почему вызываем call.answer()

Чтобы Telegram понял, что callback обработан. Иначе у пользователя может «крутиться» индикатор загрузки.

🧪 Мини-примеры callback_data (маленькие, но важные)

Мини-пример 1: единый формат

Шаблон callback_data

"faq:open"

Когда формат единый, тебе проще поддерживать проект и не искать, почему часть кнопок «не ловится».

Мини-пример 2: разбор callback_data

Разделить сущность и действие
data = "lead:start"
entity, action = data.split(":", maxsplit=1)
print(entity)  # lead
print(action)  # start

Этот приём пригодится с Дня 3+, когда сценарии станут длиннее и callback-ов станет много.

🕷️ Парсинг 200 объектов + фильтр по цене

Новая цель дня в парсере

  • Увеличить выборку до 200 объектов
  • Добавить простую фильтрацию по цене
  • Сохранить результат в CSV

Пошаговая схема обработки данных

  1. Собираем сырые строки с сайта (цены ещё «грязные»).
  2. Очищаем цену до числа.
  3. Применяем фильтр min/max.
  4. Сохраняем итог в CSV и проверяем глазами.
Функция фильтра
def parse_price(raw: str) -> int:
    digits = ''.join(ch for ch in raw if ch.isdigit())
    return int(digits) if digits else 0


def filter_by_price(rows: list[dict], min_price: int = 0, max_price: int = 10**9) -> list[dict]:
    result = []
    for row in rows:
        price = parse_price(row.get("price", "0"))
        if min_price <= price <= max_price:
            result.append(row)
    return result
Даже такой простой фильтр уже делает скрипт «бизнесовым»: клиенту редко нужны «все подряд данные».

📄 Экспорт CSV так, чтобы клиенту было удобно

Рекомендуемые поля

  • title
  • price
  • url
  • parsed_at (дата сбора)

Эти поля закрывают базовый вопрос заказчика: что нашли, сколько стоит, где это лежит и когда собрано.

Сохранение с меткой времени
from datetime import datetime

for row in rows:
    row["parsed_at"] = datetime.now().strftime("%Y-%m-%d %H:%M:%S")

with open("result.csv", "w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=["title", "price", "url", "parsed_at"])
    writer.writeheader()
    writer.writerows(rows)

Такая таблица сразу пригодна для отчёта клиенту и последующей аналитики.

Это «скелет» любого CSV-экспорта: заголовок + список словарей.

Мини-пример 2: защита от пустого списка

Проверка перед сохранением
if not rows:
    print("Нет данных для сохранения")
else:
    print(f"Записываем {len(rows)} строк")

Такой маленький guard спасает от «тихих» ошибок, когда файл создаётся, но в нём нет полезных данных.

🎥 Видео-демо 30–60 секунд (ключ к продажам)

Сценарий демо для бота

  1. Открываешь Telegram и вводишь /start.
  2. Показываешь кнопки и 2-3 нажатия.
  3. Показываешь, что бот отвечает корректно.

Сценарий демо для парсера

  1. Запускаешь скрипт.
  2. Показываешь лог «Сохранено N строк».
  3. Открываешь CSV и листаешь первые строки.

Готовый шаблон речи для демо (можешь читать почти дословно)

  • «Это Telegram-бот для приёма заявок: вот старт, вот меню, вот обработка кнопок».
  • «Это парсер: запускаю, получаю N записей, выгружаю в CSV с нужными полями».
  • «Проект можно расширять: добавить БД, уведомления, интеграции».
Клиенты покупают глазами: маленькое живое демо часто эффективнее длинного текста.

🧯 Частые ошибки Дня 2

  • Inline кнопки не работают: забыт callback handler или не подключён router.
  • Кнопки не отображаются: keyboard не передаётся в `reply_markup`.
  • Фильтр цены ломается: в цене есть символы валюты, пробелы, неочевидный формат.
  • CSV пустой: фильтр слишком строгий, либо не те CSS-селекторы.
