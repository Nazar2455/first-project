Тема: собираем parser-template, чтобы новые сайты подключались конфигом, а не переписыванием проекта с нуля.

← День 13 (выбор трека) Переключиться на БОТЫ День 14 →

1) Целевая структура parser-template

parser-template
parser-template/
├── config/
│   └── selectors.json
├── parsers/
│   └── adapter.py
├── services/
│   ├── http_client.py
│   ├── exporter.py
│   └── validation.py
├── main.py
├── requirements.txt
└── README.md

2) Идея конфигурации селекторов

selectors.json (пример)
{
  "card": ".item-card",
  "title": ".item-title",
  "price": ".item-price",
  "url": "a.item-link"
}

Меняешь сайт — чаще всего меняешь конфиг и небольшой adapter, а не всю кодовую базу.

3) Проверка качества после рефакторинга

  • Количество строк не важнее читаемости.
  • Функции короткие и предсказуемые.
  • Ошибки логируются, а не молча игнорируются.
  • README объясняет запуск за 1 минуту.

4) Практика

  1. Сделай parser-template из текущего проекта.
  2. Перекинь 2 разных сайта через конфиг/adapter.
  3. Сравни время старта нового парсера до/после.
  4. Оформи результат как mini-case в портфолио.
Это и есть коммерческий рычаг: меньше рутины → больше выполненных заказов.
